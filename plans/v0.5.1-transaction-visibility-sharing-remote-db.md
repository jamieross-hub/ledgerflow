# LedgerFlow v0.5.1 交易可见性、远程数据库与分享增强计划

> 创建日期：2026-03-07  
> 状态：已建立，待切主线（聚焦 **交易附件可见性、最后修改时间、远程 MySQL 连接、账单分享能力**）。

---

## 1. 当前版本目标

`v0.5.1` 聚焦几类非常实用、但目前还不够“产品化”的能力补完：

1. **交易列表在点击前就能看出是否有附件**
2. **交易详情修改后显示最后修改时间，增强可追溯性**
3. **增加远程 MySQL 连接能力，逐步脱离纯本地单机模式**
4. **增加账单分享能力，便于导出、转发、协作查看**

核心原则：

- 先补“可见性与可追踪性”，再扩展协作与连接能力
- 不为了分享而泄露隐私，分享内容必须可控
- 远程数据库连接优先做清晰配置与连接测试，不急着一次做成全同步平台
- 能复用现有 connection-config / database-settings 基建的地方，不重复造轮子

## 2. P0：交易可见性增强

### 2.1 交易列表增加附件标志 ✅
目标：

如果交易详情里已经有附件，用户在列表页、卡片或其他入口处，**在点击前** 就应该知道，而不是点进去才发现。

重点处理：

- 在交易列表行增加附件图标 / 徽标
- 移动端列表与桌面表格都要考虑展示方式
- 附件数量可作为增强信息（可先不强制显示）
- 不要让标志喧宾夺主，优先轻量但明显

验收标准：

- 用户在进入详情前就能识别“这笔交易有附件”
- 小屏与大屏下都不显得拥挤
- 不影响现有排序、筛选、批量操作

### 2.2 交易增加最后修改时间
目标：

让交易不只是有创建信息，还能知道“最后一次被改是什么时候”，方便追查误改、补录和 AI 修正后的变动。

重点处理：

- 在交易数据结构中增加 `updatedAt`
- 新建、编辑、AI 修正、附件变更时都更新该字段
- 详情页展示最后修改时间
- 后续可扩展为“最近改动排序”或“修改痕迹提示”

验收标准：

- 修改交易后 `updatedAt` 正常变化
- 详情页可直观看到最后修改时间
- 不破坏现有导入 / 导出 / 持久化逻辑

## 3. P0：远程数据库连接能力

### 3.1 增加远程 MySQL 连接配置入口
目标：

在现有连接配置与数据库设置基础上，增加远程 MySQL 的配置、保存与测试能力。

重点处理：

- 支持 MySQL 连接必要字段：host、port、username、password、database、ssl 等
- 连接信息支持保存、编辑、启用 / 停用
- 增加清晰的连接测试反馈
- 区分“本地模式 / 远程数据库模式”的使用边界

验收标准：

- 用户可录入并保存远程 MySQL 连接信息
- 可进行基础连接测试
- 表单和错误提示清楚，不让用户猜

### 3.2 梳理远程 MySQL 接入边界
目标：

在真正接数据库前，先明确它接管哪些数据、与本地持久化如何并存、哪些能力暂不支持。

重点处理：

- 明确远程 MySQL 与本地 store 的边界
- 区分“连接配置”“数据同步”“主存储切换”三个层次
- 明确首版先支持什么，不支持什么
- 为后续服务端 / API 层预留扩展空间

验收标准：

- 远程 MySQL 不是一句口号，而是有清晰落地边界
- 后续开发时不会把本地存储和远程库混成一团

## 4. P1：账单分享能力

### 4.1 增加账单分享入口
目标：

让用户可以把单笔账单、筛选后的账单列表或阶段汇总，快速分享出去。

重点处理：

- 支持生成可复制的分享文本 / 卡片
- 可扩展为图片分享、摘要分享、链接分享
- 分享前允许隐藏敏感字段（金额、账户、订单号等）
- 分享入口尽量靠近交易详情、账单列表或导出动作

验收标准：

- 用户能快速分享账单信息
- 分享内容结构清楚、可读性好
- 默认不泄露不该暴露的敏感信息

### 4.2 支持隐私控制与分享模板
目标：

分享不是简单复制 JSON，而是给用户一套可控模板。

重点处理：

- 提供“完整 / 脱敏 / 摘要”三类分享模板
- 可选择是否包含附件提示、备注、分类、时间等字段
- 后续可扩展为群聊分享、导出分享图

## 6. 执行原则

1. 先做交易可见性与追溯能力，再推进连接和分享
2. 远程数据库先做清晰边界与配置体验，不急着一次做全
3. 分享功能默认站在隐私保护一边
4. 每完成一项都更新计划状态并执行 `npm run build`
