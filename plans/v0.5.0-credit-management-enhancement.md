# LedgerFlow v0.5.0 AI 信贷管理增强计划

> 创建日期：2026-03-07  
> 状态：已建立，待切主线（聚焦 **AI 信贷管理能力深化、年化利率/APR 等关键指标补强、提示词与测算链路增强**）。

---

## 1. 当前版本目标

`v0.5.0` 不再停留在“识别花呗 / 分期 / 贷款信息并带去还款管理”，而是把 AI 信贷管家真正往“可分析、可测算、可决策辅助”的方向推进一步。

这一版重点不是再堆更多炫技入口，而是补齐信贷管理真正有用的判断维度：

1. **年化利率 / APR / 月利率等关键指标更完整**
2. **贷款、分期、信用账单的测算结果更可操作**
3. **AI 信贷管家提示词更专业、更稳定、更少胡猜**
4. **还款管理页对信贷信息的呈现与分析更像“管理工具”而不是普通表单**

核心原则：

- 不把信贷分析做成玄学建议，优先给可验证、可追溯的测算结果
- AI 输出要明确“已识别”“推算值”“待确认”三种状态
- 年化利率、总利息、月供、剩余成本等指标要可解释
- 信贷提示词优先减少误判、漏字段、乱补全
- 优先强化现有 repayment-management 与 AI credit 模式，不另起炉灶

补充产品原则：

- AI 不直接替用户做高风险财务结论，优先给 **测算 + 依据 + 需确认项**
- AI 对每个关键字段都应区分 **原文识别 / 规则推算 / 用户补充 / 待确认**
- AI 默认输出应 **短、准、可操作**，优先结论与下一步动作
- 能进入结构化确认态时，不应长期停留在纯聊天态
- 管理动作优先“编辑 / 确认 / 标记状态”，谨慎直接删除

## 2. P0：信贷关键指标补强

### 2.1 增加年化利率 / APR / 月利率的统一表达 ✅
目标：

让信贷管理页和 AI 信贷管家不只是显示一个 `annualRate` 字段，而是能明确区分和展示：

- 年化利率（APR / IRR 口径可区分）
- 名义年利率
- 月利率
- 日利率（如适用）

重点处理：

- 梳理现有 `annualRate` 字段语义，避免混用
- 对不同类型信贷建立统一展示口径
- 对无法直接确认口径的数据标注“推算值 / 平台口径待确认”
- 在还款管理页展示时加入解释文案，避免用户误读

验收标准：

- 用户能看懂当前利率是“平台给出的”还是“系统推算的”
- 利率显示口径一致，不再模糊
- AI 回答中能引用这些字段而不混淆

### 2.2 增加总利息 / 月供 / 剩余成本等测算字段 ✅
目标：

补齐用户真正关心的几个问题：

- 这笔贷款总共会多花多少钱？
- 每个月大概要还多少？
- 剩余成本还有多少？
- 提前还款大概能省多少？

重点处理：

- 增加总利息、预计月供、剩余总成本等衍生指标
- 对等额本息 / 等额本金 / 最低还款等常见方式给出基础测算
- 对资料不足场景输出“待补字段”而不是强算
- 为后续 AI 分析和排序提供结构化指标

验收标准：

- 还款页能展示更完整的测算信息
- AI 可以围绕这些指标给出更像样的分析
- 缺数据时不胡算

## 3. P0：AI 信贷管家提示词强化

### 3.1 强化信贷识别提示词 ✅
目标：

让 AI 在识别贷款 / 花呗 / 信用卡分期 / 消费贷时，更稳定地提取关键字段，并少说废话。

重点处理：

- 明确要求优先提取：
  - 平台 / 产品名
  - 当前应还金额
  - 总欠款 / 剩余待还
  - 还款日 / 扣款日
  - 期数 / 剩余期数
  - 每期金额
  - 利息 / 服务费 / 年化利率（如可识别）
- 明确区分：
  - 原文明确出现
  - 根据结构推测
  - 无法判断待补充
- 禁止在识别结果中凭空补全关键数字
- 优化 JSON 输出稳定性，降低格式漂移

验收标准：

- 同类账单截图下输出更稳定
- 漏字段减少，胡编字段也减少
- 结构化结果更适合直接流入还款管理页

### 3.2 强化信贷分析提示词 ✅
目标：

不只会“识别账单”，还要更会回答这些问题：

- 哪些负债最值得优先处理？
- 哪些分期实际成本偏高？
- 哪些项目更应该提前还款？
- 现金流压力主要来自哪里？

重点处理：

- 增加围绕 APR、剩余成本、还款日集中度的分析框架
- AI 输出要优先给结论，再给依据，再给行动建议
- 对不确定内容明确写“建议核对”
- 引导 AI 少做情绪化鼓励，多给排序和判断依据

验收标准：

- AI 信贷回答更像财务分析助手，而不是普通聊天机器人
- 输出结构更清楚、决策价值更高
- 提示词可复用到还款建议、风险排查、还款优先级分析

### 3.3 补齐识别结果的保存按钮与跳转按钮 ✅
目标：

当前 AI 信贷管家在识别出意图或结构化结果后，用户还缺少明确的下一步动作入口。需要补齐“保存 / 带去管理 / 跳转查看”这类按钮，让识别结果不再悬空。

重点处理：

- 对识别出的信贷项目增加显式操作区
- 至少支持两类动作：
  - 保存/带入还款管理
  - 跳转到对应管理页继续处理
- 当识别结果不完整时，按钮文案也要区分“去补充”与“去保存”
- 保持移动端操作清晰，不把按钮藏得太深

验收标准：

- 用户识别完后，能立刻知道下一步怎么做
- 不再出现“识别出来了但没有保存按钮 / 跳转按钮”的断链感
- 识别 → 管理 的路径明显变短

### 3.4 AI 信贷助手接入还款计划 / 还款账户 / 还款流水检索 ✅
目标：

让 AI 信贷助手不只会识别账单和给建议，还能直接检索当前系统里已经存在的：

- 还款计划
- 扣款/还款账户
- 交易流水中已记录的还款行为

重点处理：

- 在 AI 信贷模式中补充还款管理上下文，而不只读取普通账本快照
- 让 AI 能按“负债 -> 还款计划 -> 扣款账户 -> 已发生还款流水”串起来回答
- 对查询类问题支持直接回答：
  - 某笔贷款接下来怎么还
  - 是从哪个账户扣款
  - 最近是否已经还过 / 还了多少 / 还款记录是否缺失
- 尽量区分“计划中的应还”与“交易流水中已发生的还款”，避免混淆
- 若流水未匹配到对应负债，要明确提示“计划存在，但流水侧未确认”
- AI 信贷助手在模型流式输出阶段，应优先预输出“台账卡片 UI 框架”，实现正文分析与卡片骨架并行展示

验收标准：

- AI 信贷助手可直接回答还款计划、还款账户、历史还款流水相关问题
- 回答中能明确区分计划数据与已发生流水数据
- 用户不必手动切页也能先问清楚“这笔现在还到哪一步了”
- 流式回答过程中，用户能先看到台账卡片 UI 骨架，而不是等全文结束后才看到结构化卡片

### 3.5 优化 AI 信贷助手回答长度与多轮补充识别 ✅
目标：

让 AI 信贷助手在真实使用里更像“可连续补全的信贷助理”，而不是每次都给一大段过长回答后结束。

重点处理：

- 优化默认回答长度：
  - 识别类问题优先短结论 + 卡片
  - 分析类问题优先结论 + 2~4 条依据 + 1~3 条下一步建议
  - 避免长篇重复解释、模板化废话、把同一字段反复说多遍
- 在多轮对话中支持“补充信息后继续识别”：
  - 用户追加发一句补充说明
  - 用户再补一张图 / 再补一个字段
  - 系统应基于前文上下文继续完善同一笔信贷对象，而不是总当成全新问题
- 对“信息不完整”的识别结果，支持明确引导：
  - 还缺什么
  - 用户补什么最有效
  - 补完后如何合并成最终可保存的还款管理条目
- 当多轮补充后字段已足够时，应支持进入“最终确认 / 保存到还款管理”状态，而不是一直停留在聊天态
- 后续需与“已保存条目可编辑”能力联动，避免用户补充一次就新建一条重复负债

验收标准：

- AI 信贷回答明显更短、更聚焦，默认不再输出过长大段文字
- 用户在连续补充后，系统能延续识别同一笔信贷对象
- 补充完成后，可形成最终可保存/可带去还款管理的结果
- 不会因多轮补充而不断生成重复候选条目

### 3.6 信贷识别结果增加字段级来源 / 置信度 / 待确认状态 ✅
目标：

让 AI 信贷结果不只是“看起来像对”，而是对每个关键字段都有来源、状态与可信度，便于用户审阅和纠错。

重点处理：

- 为关键字段增加来源类型：
  - 原文识别
  - 规则推算
  - AI 推断
  - 用户补充
- 为关键字段增加状态：
  - 已确认
  - 待确认
  - 低置信度
- 支持展示字段证据：来自哪段文本、哪张图片、哪一轮补充
- 增加字段更新时间与最近修改来源（AI / 用户）
- 在卡片与保存前确认态中，统一展示这些元信息

验收标准：

- 用户能看懂每个关键字段“为什么是这个值”
- AI 结果出错时可回溯，不再像黑盒猜测
- 保存前可快速识别哪些字段仍需人工确认

### 3.7 AI 信贷识别结果增加“保存前确认态”与冲突分流 ✅
目标：

让“保存到还款管理”前多一个轻量但明确的确认层，减少误保存、误合并、误覆盖。

重点处理：

- 当字段已基本足够时，从聊天态切入结构化确认态
- 在确认态中集中展示：
  - 产品名
  - 当前应还
  - 剩余待还
  - 还款日
  - 每期金额
  - APR / 年化
  - 扣款账户
  - 数据状态（明确 / 推测 / 待确认）
- 提供明确动作：
  - 确认保存
  - 我再补一个字段
  - 这不是同一笔，拆成新卡片
- 当系统检测到可能与已有草稿 / 已保存负债冲突时，支持用户选择“合并 / 新建 / 仅更新部分字段”

验收标准：

- 用户保存前能快速做一次结构化校验
- 多轮补充后的结果不会轻易误保存成错误负债
- 遇到相似项目时能显式分流，而不是静默合并

### 3.8 多轮补充识别增加对象绑定与补全进度 ✅
目标：

让 AI 在连续多轮补充中，明确知道“当前正在补的是哪一笔信贷”，并向用户持续展示补全进度。

重点处理：

- 为待确认信贷对象引入会话级草稿标识，例如：
  - `draftCreditId`
  - `matchReason`
  - `lastConfirmedFields`
  - `lastMissingFields`
  - `mergedFromTurns`
- 在每轮补充后更新：
  - 当前绑定对象
  - 已补字段数量
  - 仍缺字段列表
- 在 UI 上明确展示：
  - 当前补充对象名称
  - 已补字段进度（如 3/7）
  - 当前最建议补充的 1~2 个字段
- 降低“同名产品 / 多笔分期”被错误绑定到同一对象的概率

验收标准：

- 用户能清楚知道当前补的是哪一笔
- 多轮补充不再轻易跑偏或越补越乱
- 同一笔负债在多轮中会持续完善，而不是频繁重新生成

### 3.9 计划应还 vs 实际已还增加差异解释与缺口提示 ✅
目标：

让 AI 不只会分别展示“计划中的应还”和“流水中已还”，还要能解释两者的差额与缺口原因。

重点处理：

- 对每笔负债输出：
  - 本期计划应还
  - 实际已还
  - 当前差额
  - 差额原因（待确认 / 部分还款 / 未匹配到账单 / 跨账户还款等）
  - 数据可信度
- 明确区分：
  - 计划存在但流水未匹配
  - 流水存在但未归属到具体负债
  - 部分已还但未结清
- 在 AI 回答中优先输出“是否有缺口”“缺口有多大”“下一步建议核对什么”
- 为后续的汇总问答与风险标签复用差异解释结构

验收标准：

- 用户能快速知道每笔负债当前到底还差多少
- AI 能解释“为什么计划和实际对不上”
- 回答更像台账管家，而不是两堆数据拼接

### 3.10 补齐高频专业问答模板与回答框架 ✅
目标：

明确 AI 信贷助手在真实使用中要重点支持哪些高频问法，并为这些问法建立稳定回答框架。

重点处理：

- 补齐高频问题模板：
  - 风险判断类：哪些最危险 / 哪些成本最高 / 哪些适合提前还
  - 现金流类：本月总应还多少 / 如果只能多拿 2000 元先补哪几笔
  - 复盘类：上个月总共还了多少 / 哪些还款未关联负债
  - 决策类：提前结清 A 还是先补 B 更划算
- 为每类问题约束回答结构：
  - 先结论
  - 再依据
  - 再建议动作
  - 最后列待确认项（若有）
- 尽量避免 AI 每次自由发挥导致话术漂移
- 为后续回归测试预留典型问答样例集

验收标准：

- 同类问题回答结构稳定，质量更可控
- 更像专业信贷助理，而不是泛化聊天模型
- 便于后续做自动化测试与回归对比

## 4. P1：还款管理页继续增强

### 4.1 增加提前还款 / 还款策略模拟 ✅
目标：

让用户能快速比较不同还款策略的影响，例如：

- 本月多还 1000 元会怎样
- 优先还高利率 vs 优先还小余额
- 提前结清某笔贷款大约可省多少

### 4.2 增加信贷风险标签与解释 ✅
目标：

把“高利率 / 还款日过于集中 / 压力过高 / 信息缺失”这些风险，用更直接的标签方式呈现给用户。

### 4.3 负债列表管理支持编辑已保存条目 ✅
目标：

让用户对已经保存到“负债管理”的条目，不必删除重建，而是可以直接编辑修正，例如：

- 修改负债名称
- 修正年化利率 / 期数 / 总还款
- 补充还款日 / 账单日 / 扣款账户
- 修正记录方式 / 宽限期 / 备注型信息

重点处理：

- 在负债列表 / 台账预览中增加显式“编辑”入口
- 支持对已保存条目进行原位编辑或侧边编辑
- 编辑后重新联动：
  - 利率与月供测算
  - 风险标签
  - 提前还款模拟
  - AI 信贷助手读取到的还款上下文
- 避免把“编辑已保存条目”做成重新新增一条，防止重复数据

验收标准：

- 用户可直接修改已保存负债，而无需先删除再新增
- 编辑后各类测算/风险/AI 上下文立即刷新
- 不产生重复负债条目

### 4.4 AI 信贷助手支持负债 / 账单 / 还款记录管理动作 ✅
目标：

让 AI 信贷助手不只会识别和建议，还能围绕还款管理做更直接的结构化动作。

重点处理：

- 支持负债动作：
  - 新增负债
  - 编辑负债
  - 删除负债
  - 标记已结清
  - 标记暂缓处理
- 支持账单动作：
  - 新增本期应还
  - 修改本期应还
  - 删除错误账单
  - 标记已还 / 部分已还 / 延期
- 支持还款记录动作：
  - 新增还款记录
  - 删除误匹配流水
  - 关联到某笔负债
  - 指定扣款账户
- 明确哪些动作可直接执行，哪些必须二次确认
- 与 3.7 保存前确认态联动，避免操作越做越乱

验收标准：

- 用户可在聊天中完成更多管理动作，而不只是看建议
- 删除 / 覆盖 / 标记类操作边界清晰，不易误触
- AI 与还款管理页之间的数据闭环更完整

### 4.5 AI 信贷助手支持关键汇总问答与执行入口 ✅
目标：

让 AI 信贷助手能稳定回答高频汇总问题，并在可执行时给出进一步处理入口。

重点处理：

- 支持直接回答：
  - 当前总欠款
  - 本月总应还
  - 已还多少
  - 还差多少
  - 总利息 / 剩余利息 / 剩余总成本
  - 最低还款合计
  - 本周 / 本月即将到期项目
- 汇总时明确区分：
  - 负债台账值
  - 计划账单值
  - 实际还款流水值
  - 推算汇总值
- 对关键汇总问题给出相应后续动作：
  - 去补记录
  - 去确认差额
  - 去查看高风险项
  - 去进入保存前确认态
- 与 3.9 差异解释层共享同一套数据口径

验收标准：

- 用户问“本月应还多少 / 还差多少 / 总利息多少”时能快速得到稳定回答
- 回答口径一致，不会混淆台账、计划、流水和推算值
- 汇总结果可继续驱动后续管理动作

### 4.6 已结清 / 已关闭 / 暂缓处理状态管理
目标：

让真实世界中的负债生命周期管理更完整，而不是只有“存在”与“删除”两种状态。

重点处理：

- 为负债与账单增加状态：
  - 进行中
  - 已结清
  - 已关闭
  - 暂缓处理
- 在列表、风险提示、汇总与 AI 检索中体现状态差异
- 已结清项目保留历史，但默认降权展示
- 暂缓处理项目不应继续被默认推荐为优先处理对象
- 删除与状态变更要有清晰边界：能标状态就尽量不删历史

验收标准：

- 用户不必靠删除来清理历史项目
- AI 在分析优先级时能识别哪些项目已结清或暂缓
- 台账长期使用后仍保持清晰可管理

## 6. 执行原则

1. 先统一字段语义，再强化 AI 提示词
2. 先让测算结果可靠，再让建议更智能
3. 所有 AI 输出都要区分“识别值 / 推算值 / 待确认”
4. 每完成一项都更新计划状态并执行 `npm run build`
5. 所有新增管理动作都要定义确认边界与回滚思路
6. 对关键问答与识别链路，逐步补齐可回归测试样例
