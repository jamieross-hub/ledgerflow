# LedgerFlow v0.4.5 UI / UX 动态优化计划

> 创建日期：2026-03-06  
> 状态：下一阶段主线候选（以 **动态 CSS、响应式重排、页面动效与交互舒展度优化** 为重点）。

---

## 1. 当前版本目标

`v0.4.5` 主要不追求大功能扩张，而是补齐上一阶段已经暴露出来的 **UI / UX 动态感不足、布局不够舒展、页面缺少响应式层次变化** 的问题。

核心方向：

1. 动态 CSS 响应与页面动画补强
2. AI 助手页的宽屏 / 窄屏自适配重构
3. 让状态信息、辅助信息、最近交易提示回到“按需出现”，不要长期挤占主界面
4. 统一页面进入、切换、展开、悬浮、反馈的动态语言
5. 在不牺牲交互效率的前提下提升“页面是活的”这种感受

核心原则：

- 动画服务信息层次，不做无意义飘动
- 宽屏强调展开与左右呼应，窄屏强调收紧与留白
- 动效必须轻量，不能拖慢页面
- 优先改“用户一直看得到”的主流程页面
- 保持 LedgerFlow 的专业感，不做花哨炫技

## 2. P0：AI 助手页面重构优化

### 2.1 助手欢迎卡片组宽屏左右对齐 / 窄屏收紧 ✅
目标：

让“你好，我是你的财务助手”及其关联卡片组在 PC 宽屏时具备更舒展的左右布局，在窄屏时自动收紧，不再一直维持单列堆叠。

重点处理：

- 宽屏状态下按信息层级拆成左右区块
- 保持主介绍、今日要做、本月总结、主动洞察之间的主次关系
- 中等宽度时平滑退化，不突然断层
- 手机端继续以单列为主，但减少不必要高度

验收标准：

- 宽屏下卡片组布局更舒展
- 中屏到窄屏过渡自然
- 不因响应式改造造成内容跳位或阅读断裂

### 2.2 语义召回信息改为按需显示 ✅
目标：

不要让语义召回信息强制常驻在 AI 助手主界面，占据首屏空间。

重点处理：

- 语义召回信息默认折叠、弱化或迁移到次级区域
- 仅在用户需要理解上下文来源时再展开
- 保留排障与解释能力，但不打断主对话区

验收标准：

- AI 助手主界面首屏更干净
- 语义召回仍可被查看
- 不影响上下文理解与调试能力

### 2.3 最近一笔交易改为动态出现在输入区 ✅
目标：

不要把“最近一笔交易”固定压在输入框上方，而是改为在输入区内部或输入上下文中动态出现。

重点处理：

- 作为输入提示、草稿提示或可收起提示条融入输入区
- 空闲时可见，输入中自动弱化或收起
- 避免持续占据输入区上方高度

验收标准：

- 最近交易信息仍可被快速感知
- 输入区视觉更轻
- 手机上输入空间更宽松

---

### 2.4 AI 记账助手页增加最近三条记账时间轴 ✅
目标：

在 AI 记账助手页补充一个轻量时间轴区域，只展示最近三条记账记录，用来增强“刚刚记了什么”的上下文感。

重点处理：

- 时间轴仅显示最近三条，避免信息过载
- 展示时间、金额、分类/备注等最关键字段
- 视觉上与助手页欢迎区、输入区保持同一语言
- 窄屏下保持紧凑，不额外挤压输入与消息区
- 后续可与“最近一笔交易”输入区提示形成联动

验收标准：

- 助手页可看到最近三条记账记录
- 时间轴结构清楚，不显拥挤
- 手机端也能稳定显示


### 3.1 页面进入 / 区块出现 / 卡片悬浮的统一动效语言 ✅
目标：

建立 LedgerFlow 的轻量动态语言，让页面切换、卡片出现、数据模块加载时更自然。

重点处理：

- 页面级进入动画统一节奏
- 卡片 hover / focus / active 状态统一反馈
- 面板出现、收起、切换时减少“硬切”感
- 保持性能安全边界

验收标准：

- 页面更有动态感但不浮夸
- 动效节奏统一
- 不出现明显卡顿

### 3.2 响应式断点下的布局过渡动画 ✅
目标：

让页面在宽屏 / 中屏 / 窄屏切换时不显得生硬，而是有更自然的收放感。

重点处理：

- 卡片栅格变化更平滑
- 常见信息区支持柔和过渡
- 减少断点切换时的突兀错位感

验收标准：

- 响应式切换更自然
- 布局变化不再像“突然重排”

### 3.3 图表、提示条、空状态的轻量动态增强 ✅
目标：

给图表、提示条、空状态补上恰当的轻动态，提升页面生命感。

重点处理：

- 图表加载与刷新时的轻微过渡
- 提示条出现 / 消失的统一节奏
- 空状态与骨架状态更自然

验收标准：

- 页面状态切换更顺滑
- 不抢内容主体风头

## 4. P1：首页与主要页面 UI / UX 优化记录

### 4.1 首页欢迎区动态化与主题风格统一
目标：

把首页欢迎区继续打磨成更像正式首页 Hero 的区域，增强主题色背景、文字重心与轻动态表达。

### 4.2 Dashboard / Transactions / Assistant 三大页面动态语言统一
目标：

统一三大核心页面的卡片、按钮、状态区、图表与过渡反馈，不再各说各话。

### 4.4 页面文案语气优化：更自然、略带幽默、允许轻动画点缀 ✅
目标：

让首页、帮助页、AI 助手页等核心页面的文案从“功能说明”进一步走向“更像产品本身会说话”，语气更自然，允许少量幽默感，并可结合轻动画元素增强记忆点。

重点处理：

- 保持专业感，不做过头的玩梗
- 欢迎语、空状态、帮助提示、助手引导文案可适度变得更有人味
- 与轻量动态元素配合，但不喧宾夺主
- 文案风格在不同页面之间保持同一人格

验收标准：

- 页面文案更自然，不再全是工具说明口吻
- 允许轻微幽默，但不会破坏可信度
- 与页面动态风格形成统一体验

### 4.5 WebDAV 下载恢复前提供版本选择
目标：

当用户点击 WebDAV 下载并恢复按钮时，不再默认直接恢复最新备份，而是先弹出可恢复版本列表供用户选择。

重点处理：

- 点击“下载并恢复”后，先拉取可用备份版本列表
- 弹窗中展示版本时间、文件名、相对新旧顺序等关键信息
- 用户明确选择后再执行下载与恢复
- 兼容 WebDAV 不支持完整列表能力时的降级策略
- 避免误恢复，提升恢复操作可控性

验收标准：

- 恢复前可看到可选版本列表
- 用户可明确选择某个版本执行恢复
- 不支持版本列表时有清晰降级提示
- 恢复流程比当前更可控、更可解释

## 5. 建议执行顺序

### Sprint A（Assistant 优先）
1. 助手欢迎卡片组宽屏左右对齐 / 窄屏收紧
2. 语义召回按需显示
3. 最近一笔交易融入输入区

### Sprint B（动态 CSS）
4. 页面 / 卡片 / 面板统一动效语言
5. 响应式布局过渡动画
6. 图表 / 提示条 / 空状态动态增强

### Sprint C（全站统一）
7. 首页欢迎区动态化与主题统一
8. Dashboard / Transactions / Assistant 三大页面动态语言统一
9. 交互反馈补强

---

## 6. 执行原则

1. 每项优化都需要同时验证宽屏与窄屏
2. 动效优先使用轻量 CSS 方案，谨慎增加复杂逻辑
3. 首先服务可读性与层次，再服务“看起来动态”
4. 不让辅助信息长期霸占主界面
5. 每完成一项都更新计划状态，并执行 `npm run build`
