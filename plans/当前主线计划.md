## 2. 主线任务池

## 2.1 P0（当前优先）

### 0.5.7 UI 审查结论（2026-04-26）

本轮审查后，下一版 `v0.5.7` 优先处理以下问题：

1. 首页、助手页、负债页都存在“说明太多、首屏动作不够聚焦”的问题
2. 助手页、交易页、分类页在移动端存在明显的控件拥挤、按钮挤压或横向注意力分散
3. 空状态、引导卡片、调试入口之间缺少优先级管理，导致首屏经常不是“最该做的事”

### A. v0.5.7 首屏与引导重排

- [x] 首页收敛欢迎 Banner 与首次引导卡片
  - 目标：避免 `DashboardPage` 首屏同时出现“欢迎 Hero + 新手引导 + 调试入口”三套注意力中心
  - 动作：把首次引导压缩为单条提示或可折叠入口；保留一个主 CTA，不再让首页上半屏被说明卡占满
  - 结果：首次引导仅在无交易数据时展示，并压缩为 1 条建议 + 2 个动作；调试 FAB 同步移出非开发态
  - 参考：`src/pages/dashboard/DashboardPage.tsx`、`src/features/dashboard/components/DashboardWelcomeBanner.tsx`

- [ ] 财务分析页空状态改为“更短说明 + 更强下一步”
  - 目标：避免大块留白只承载一句空状态，提升从“没有数据”到“去补数据”的转化效率
  - 动作：减少垂直空白，增加最近动作建议或最低样本提示，保留主次 CTA 但让主动作更突出
  - 参考：`src/pages/financial-analysis/FinancialAnalysisPage.tsx`、`src/shared/ui/EmptyState.tsx`

- [x] 负债页空状态从说明墙改为行动卡
  - 目标：避免“先补齐基础信息，再生成策略”区域在无数据时过度占据首屏
  - 动作：将 3 步长文案改为 1 个主动作 + 2 个次动作 + 简短说明，优先露出“新增负债 / 上传账单”
  - 结果：首屏改为行动卡，优先露出“上传账单自动识别 / 手动添加负债”，有数据时不再展示空壳指标区
  - 参考：`src/pages/repayment-management/RepaymentManagementPage.tsx`

### B. v0.5.7 移动端可用性修复

- [x] 修复助手页移动端 topbar 挤压与控件溢出
  - 目标：解决 `AssistantPage` 在窄屏下模式切换、模型选择、快捷动作同时出现时的错位与遮挡
  - 动作：重排 topbar 分层；减少同层操作数；为移动端提供更轻量的操作入口
  - 结果：移动端 topbar 改为分层布局，并隐藏低优先级的宽屏切换按钮
  - 参考：`src/pages/assistant/AssistantPage.tsx`、`src/app/styles/assistant.css`

- [x] 修复交易页移动端筛选区的操作密度
  - 目标：让“新增 / 批量 / 洞察 / 隐私 / 筛选”形成清晰主次，而不是一排并列抢注意力
  - 动作：保留一个首要 CTA，把二级动作收入口袋区或分段区；同时统一交易语义文案
  - 结果：首屏保留“记一笔 + 查看洞察”，批量/隐私收进“更多筛选与操作”，移动端不再横向滚动抢注意力
  - 参考：`src/features/transactions/components/TransactionFilters.tsx`、`src/app/styles/pages.css`

- [ ] 修复分类与标签管理页移动端行内操作拥挤
  - 目标：避免分类/标签行同时出现多个按钮与下拉，导致列表阅读和操作都变得吃力
  - 动作：将高频动作前置，低频动作折叠到更多菜单或二级行；重新梳理“计数 / 合并 / 删除 / 排序”的布局优先级
  - 参考：`src/pages/categories-accounts/CategoriesAccountsPage.tsx`、`src/app/styles/global.css`

### C. v0.5.7 页面层级与文案收口

## 3. 页面改动规范流程

后续任何页面类需求，都按下面流程推进。

### 3.1 页面规划前

1. 先明确页面职责
   - 这是“操作页”“分析页”“管理页”还是“承载页”
   - 不允许一个页面同时承担太多互相冲突的角色

2. 先明确入口与落点
   - 路由落在哪
   - 桌面侧边栏是否需要入口
   - 移动端抽屉是否需要入口
   - 是否要和已有页面形成跳转闭环

3. 先明确复用策略
   - 优先复用现有 `model`、现有组件、现有样式域
   - 禁止为临时页面直接复制整段旧页面逻辑

### 3.2 页面实现中

1. 页面层只负责装配，不堆大段业务计算
2. 可抽出的计算放到对应 `model` 文件
3. 新样式优先进入对应页面域样式文件，不再直接堆入 `src/app/styles/global.css`
4. 页面上的每个主要结论或提示，都要能说明来源，避免“像 AI 编的”
5. 页面按钮文案要直接指向动作，不用模糊表达

### 3.3 页面验收前

每次页面改动至少完成以下检查：

- [ ] 桌面端可用
- [ ] 移动端可用
- [ ] 不出现明显横向溢出 / 按钮挤压 / 文案截断
- [ ] 入口、标题、页面内容三者语义一致
- [ ] 至少完成一次 `npm run build`
- [ ] 若改动高风险链路，补最小测试或手动验收记录

### 3.4 页面完成后

1. 更新本文件中的任务状态
2. 将完成事项追加到 `plans/已完成汇总.md`
3. 若页面新增正式入口，需同步检查：
   - 路由表
   - 懒加载导出
   - 桌面导航
   - 移动端导航
   - 文案与帮助说明

## 4. 版本发布规范流程

后续每个版本发布，统一按下面流程执行。

## 4.1 发版前：版本收口

- [ ] 明确本次版本号
- [ ] 明确本次版本范围：只写真正要发的内容，不混入未来事项
- [ ] 确认主线文件中的对应任务状态已更新
- [ ] 将“已做完但还没归档”的事项追加到 `plans/已完成汇总.md`
- [ ] 确认页面入口、标题、版本文案已同步

## 4.2 发版前：质量检查

- [ ] `npm run build` 通过
- [ ] 若有关键测试，确保测试通过
- [ ] 至少手工回归以下高频页面：
  - 首页 `src/pages/dashboard/DashboardPage.tsx`
  - 交易页 `src/pages/transactions/TransactionsPage.tsx`
  - 助手页 `src/pages/assistant/AssistantPage.tsx`
  - 本次版本重点页面
- [ ] 检查移动端至少一轮
- [ ] 检查深浅主题至少一轮

## 4.3 发版前：文档同步

- [ ] 更新版本说明索引
- [ ] 如需，对应补充 `release-notes/` 下的版本说明与检查清单
- [ ] 确保计划目录中的状态与实际代码状态一致
- [ ] 不允许出现“代码已完成但计划仍未完成”或“计划写已完成但代码未落地”

## 4.4 发版执行

- [ ] 打 tag / 准备发布物
- [ ] 若本次涉及 Docker，确认镜像构建与推送动作已执行
- [ ] 若本次涉及 GitHub Release，确认发布说明完整
- [ ] 若本次涉及 PWA / 静态资源缓存，确认版本资源已更新

## 4.5 发版后

- [ ] 回填发布结果到版本说明或发布记录
- [ ] 将已完成版本从主线中移出，只保留仍未完成的事项
- [ ] 校验线上页面版本信息、入口与关键链路是否一致
- [ ] 如发现发版遗漏，优先补文档与流程，再补救动作
