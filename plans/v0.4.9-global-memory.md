# LedgerFlow v0.4.9 用户全局记忆与偏好落库计划

> 创建日期：2026-03-07  
> 状态：已完成并发布（聚焦 **用户全局记忆、偏好沉淀、多轮对话抽取、独立记忆页落库与管理**）。

---

## 1. 当前版本目标

`v0.4.9` 计划让 LedgerFlow 不只是“本轮会话里聪明一点”，而是能在多轮对话之后，逐步记住用户的稳定偏好，并把这些记忆沉淀为可查看、可编辑、可删除的长期资料。

目标不是做一个神神叨叨的黑箱记忆系统，而是做一个：

1. **可解释** 的用户偏好记忆层
2. **可控** 的长期记忆落库机制
3. **可见** 的独立页面管理入口
4. **可复用** 于 AI 助手、记账建议、分类推荐、提醒策略等场景的全局上下文

核心原则：

- 不记“所有聊天”，只记对产品有用、相对稳定的用户偏好
- 记忆必须可审阅、可编辑、可删除
- 自动抽取要保守，宁可少记，不要乱记
- 全局记忆是“长期偏好层”，不是聊天记录备份
- **记忆功能依托嵌入与重排序模型：嵌入负责召回，重排序负责选优与注入顺序**

## 2. P0：记忆模型与落库结构

### 2.1 设计用户全局记忆数据结构 ✅
目标：

先定义清楚“什么算记忆、什么不算记忆”，避免后续把一切都堆进数据库。

重点处理：

- 区分记忆类型：
  - 用户偏好（如回答风格、关注重点）
  - 账务习惯（如常用分类、常用账户、常见记账口径）
  - 风险偏好（如偏保守、重视还款提醒）
  - 展示偏好（如喜欢简洁回答、喜欢先结论后展开）
- 每条记忆建议包含字段：
  - id
  - title
  - content
  - type
  - source（来自哪类对话/行为）
  - confidence
  - status（active / archived）
  - createdAt / updatedAt
- 预留手动置顶、手动禁用、来源追溯能力
- 为嵌入 / 重排序链路预留字段：
  - embeddingText（用于向量化召回）
  - lastUsedAt（用于注入衰减与排序）
  - sourceTrace / sourceIds（用于来源追踪）
  - origin（manual / extracted / inferred）
  - pinned / disabled / score 等控制字段

验收标准：

- 有清晰稳定的数据模型
- 能区分长期偏好与临时上下文
- 后续页面与 AI 调用都能直接复用

### 2.2 增加记忆存储与基础 CRUD ✅
目标：

将全局记忆真正落入本地存储层，形成独立于聊天记录的长期数据区。

重点处理：

- 在 store / persistence 层新增 memory 数据区
- 支持新增、修改、删除、归档、启用/禁用
- 支持后续按类型筛选与排序
- 与普通聊天历史彻底分开存储

验收标准：

- 记忆可独立落库并持久化
- 刷新页面后仍能保留
- 不污染现有聊天缓存结构

## 3. P0：多轮对话后的偏好抽取

### 3.1 增加多轮对话后的记忆提炼机制 ✅
目标：

在 AI 助手完成若干轮有效对话后，尝试把“高价值、相对稳定”的用户偏好提炼出来，而不是手工维护。

重点处理：

- 仅在满足阈值时触发（如最近 N 轮对话、出现重复偏好）
- 优先抽取这些内容：
  - 用户喜欢的回答风格
  - 用户重点关心的财务维度
  - 常见表述习惯 / 记账习惯
  - 还款/预算/分类上的固定偏好
- 抽取结果先作为“候选记忆”或低风险自动写入
- 明确避免抽取：
  - 单次偶发情绪
  - 模糊猜测
  - 纯临时任务信息

验收标准：

- 能从多轮对话中提炼出少量有效偏好
- 不会一聊几句就乱记
- 抽取逻辑可解释、可调优

### 3.2 AI 助手在回答时接入全局记忆上下文 ✅
目标：

让全局记忆不是摆设，而是能真正影响后续回答风格与建议方式。

重点处理：

- 在 assistant / credit 等模式里，按需拼接全局记忆摘要
- 控制注入长度，避免 prompt 膨胀
- 优先用于：
  - 回答风格调整
  - 优先级判断
  - 建议排序
  - 跟进问题个性化
- 区分“长期偏好”与“当前会话上下文”

验收标准：

- AI 回答能体现已记住的用户偏好
- 不会因记忆注入导致回答跑偏
- 记忆上下文长度可控

## 4. P0：独立记忆页

### 4.1 新增“全局记忆”独立页面 ✅
目标：

给用户一个明确页面查看系统记住了什么，而不是把记忆偷偷藏起来。

重点处理：

- 新增独立路由与页面
- 支持按类型查看记忆
- 列表展示标题、内容、来源、更新时间、状态
- 支持查看“为什么记住这条”或来源说明

验收标准：

- 用户能直观看到当前全局记忆
- 记忆列表结构清晰，不像调试页
- 页面风格与当前产品一致

### 4.2 支持手动编辑 / 删除 / 归档记忆 ✅
目标：

让用户拥有最终控制权，避免系统记错了却没法改。

重点处理：

- 记忆项可编辑
- 可删除或归档
- 可停用单条记忆，避免继续参与 AI 上下文
- 对高风险记忆支持显式确认或撤销

验收标准：

- 用户能直接管理自己的长期记忆
- 错误记忆能方便修正
- 记忆系统透明可控

## 5. P1：后续扩展方向

### 5.1 记忆与分类推荐 / 还款建议联动
目标：

让全局记忆不仅影响聊天风格，也能影响实际产品建议。

可扩展方向：

- 常用分类优先推荐
- 默认回答深度适配
- 还款提醒策略个性化
- 预算建议按用户习惯调整

### 5.2 候选记忆审核机制
目标：

后续可增加“候选记忆池”，让系统先建议，再由用户确认是否写入长期记忆。

## 6. 建议执行顺序

### Sprint A（模型）
1. 设计用户全局记忆数据结构
2. 增加记忆存储与基础 CRUD

### Sprint B（抽取）
3. 增加多轮对话后的记忆提炼机制
4. AI 助手接入全局记忆上下文

### Sprint C（页面）
5. 新增全局记忆独立页面
6. 支持手动编辑 / 删除 / 归档

---

## 7. 执行原则

1. 先做可控落库，再做自动抽取
2. 先做可见可管，再做更智能的自动记忆
3. 所有记忆都必须支持删除与停用
4. 每完成一项都更新计划状态并执行 `npm run build`
