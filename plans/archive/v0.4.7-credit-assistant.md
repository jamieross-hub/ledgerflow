# LedgerFlow v0.4.7 AI 信贷助手与抽屉交互优化计划

> 创建日期：2026-03-06  
> 状态：已建立，待切主线（聚焦 **账单详情抽屉交互增强、AI 助手扩展信贷模式、贷款/花呗/分期识别与还款管理联动**）。

---

## 1. 当前版本目标

`v0.4.7` 计划在不打乱现有 Assistant / Transactions / Repayment 主链路的前提下，补上两条新体验线：

1. 账单详情抽屉交互更顺手，支持顶部拖拽拉伸
2. AI 助手新增一个聚焦贷款 / 花呗 / 分期 / 信用账单的独立模式
3. 让 AI 不只是“记一笔”，还能帮用户梳理“我到底欠了什么、什么时候该还、哪些是分期、哪些是贷款”
4. 尽量和现有还款管理页联动，避免新模式变成孤岛

核心原则：

- 先把复杂信贷信息整理清楚，再谈自动化
- 不把普通记账助手搞成全能大杂烩，信用/贷款场景要有独立入口
- 抽屉交互优先服务信息查看效率，不做花哨拖拽
- 新模式优先“识别 + 整理 + 引导”，再逐步走向“写入 + 管理”

---

## 2. P0：账单详情抽屉交互优化

### 2.1 账单详情抽屉顶部支持拉伸 ✅
目标：

让交易详情抽屉在打开后，支持通过顶部区域拖拽调整高度，提升长内容、附件、时间轴和账单信息查看体验。

重点处理：

- 抽屉顶部增加可拖拽拉伸的交互区
- 支持在合理最小/最大高度之间调整
- 鼠标端与触控端都要尽量自然
- 不影响当前抽屉打开/关闭、滚动和移动端表现
- 保持默认高度依然适合普通详情查看

验收标准：

- 用户可通过抽屉顶部调整高度
- 长内容查看更顺手
- 不出现拖拽卡顿、误触关闭、滚动冲突

## 3. P0：AI 助手扩展“AI 信贷管家”模式

### 3.1 AI 助手页增加“AI 信贷管家”胶囊按钮 ✅
目标：

在现有“AI 记账 / AI 助手”基础上，新增第三个模式入口，暂定命名为 **AI 信贷管家**。

命名说明：

- 不建议直接叫“AI 信贷”，太像能力标签
- “AI 信贷管家”更像一个独立工作模式，语义更清楚
- 后续也可根据页面气质再评估为“AI 还款管家 / AI 负债助手”

重点处理：

- 在 Assistant 页面顶部模式切换区新增第三个胶囊按钮
- 新模式与普通 AI 助手 / AI 记账清晰区分
- 页面文案、预设提问、欢迎区内容转向信贷 / 花呗 / 分期 / 贷款管理语境
- 不挤压现有模式的主体验

验收标准：

- Assistant 页可切换到 AI 信贷管家模式
- 模式之间职责明确，不会混淆
- 信贷模式在视觉和内容上有独立感

### 3.2 AI 信贷管家支持贷款 / 花呗 / 分期信息识别 ✅
目标：

让用户通过文本、截图、账单截图等方式，把贷款、花呗、分期付款、信用账单中的关键信息交给 AI 整理出来。

重点处理：

- 识别内容包括但不限于：
  - 平台 / 产品名（如花呗、借呗、银行分期、消费贷等）
  - 当前应还金额
  - 总欠款 / 剩余待还
  - 还款日 / 扣款日
  - 期数 / 剩余期数
  - 每期金额 / 利息（如可识别）
- 支持从截图 / 账单中提炼结构化信贷信息
- 输出结果应适合后续进入还款管理页，而不是只给一段解释文本
- 对信息缺失情况明确标记“待补充”

验收标准：

- 可识别至少一类典型信贷/分期截图并提炼关键信息
- 结果结构清楚，便于后续录入或联动
- 不把普通消费交易误判成贷款/分期

### 3.3 AI 信贷管家联动还款管理页 ✅
目标：

让 AI 信贷管家模式不只是“识别一下就算了”，而是能把结果自然衔接到还款管理页面的已有能力中。

重点处理：

- 识别结果可引导用户进入还款管理页
- 后续可支持将识别出的贷款/分期信息预填到还款管理结构中
- 对花呗、分期付款、信用卡账单等半结构化负债，建立统一过渡层
- 尽量复用现有 repayment-management 的字段和逻辑，避免另起一套债务体系

验收标准：

- 信贷助手与还款管理不是两张皮
- 用户能从识别结果顺畅进入管理流程
- 新模式能强化现有还款页，而不是替代它

### Sprint A（抽屉交互）
1. 账单详情抽屉顶部支持拉伸

### Sprint B（AI 信贷模式建模）
2. 增加 AI 信贷管家模式入口
3. 补欢迎区 / 预设问题 / 模式文案
4. 明确信贷识别结果结构

### Sprint C（识别与联动）
5. 支持贷款 / 花呗 / 分期识别
6. 结果联动还款管理页

---

## 5. 执行原则

1. 先做低风险 UI / 模式切换，再推进结构化识别
2. 信贷模式优先增强现有 repayment-management，不重复造系统
3. 所有识别结果都要区分“已确认”与“待补充”
4. 每完成一项都更新计划状态并执行 `npm run build`
