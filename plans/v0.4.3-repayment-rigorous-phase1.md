# 还款管理严谨化改造：第一步分析（2026-03-06）

## 目标

梳理当前 `RepaymentManagementPage` 已有结构、字段能力与关键缺口，为后续“严谨版还款管理”设计打基础。

---

## 一、当前已有能力

根据 `src/pages/repayment-management/RepaymentManagementPage.tsx` 当前实现，已具备以下基础：

### 1. 负债基础字段（部分）
当前可录入或维护的字段包括：

- `name`：负债名称
- `type`：负债类型（信用卡 / 消费贷 / 贷款）
- `balance`：剩余本金/余额
- `annualRate`：年化利率（贷款场景）
- `remainingMonths`：剩余期数（月）
- `totalPeriods`：总期数
- `paidPeriods`：已还期数
- `loanPrincipal`：借款金额
- `totalRepayment`：总还款金额
- `billDay`：账单日（部分类型）
- `repaymentDay`：还款日

### 2. 部分计算能力
已存在：

- 负债健康度计算
- 最低还款额估算
- 还款优先级排序
- 部分年化利率推断逻辑
- AI 还款建议生成
- 基于账单截图识别负债信息

### 3. 页面层已有输出
当前页面已有：

- 负债概览
- 优先还款建议
- 还款日提醒
- AI 建议文本

---

## 二、当前关键缺口

虽然已有基础，但从“严谨台账”的要求看，当前缺口很明显。

### 1. 缺少“还款计划层”
当前更像是在管理“债务条目”，而不是管理“逐期还款计划”。

缺少：

- 每一期应还记录
- 每期应还日期
- 每期应还本金
- 每期应还利息
- 每期应还总额
- 每期状态（待还 / 已还 / 部分还 / 逾期）

### 2. 缺少“实际还款记录层”
当前没有形成严谨的还款记录体系。

缺少：

- 实际还款时间
- 实际还款金额
- 实际扣款账户
- 关联交易 ID
- 还款方式（手动 / 自动识别 / 自动扣款）
- 部分还款 / 提前还款 / 撤销记录

### 3. 扣款账户未闭环
用户要求关注“还款后扣款账户”，但当前结构中尚未形成完整模型。

缺少：

- 默认扣款账户
- 实际扣款账户
- 账户切换记录
- 扣款后账户余额联动校验
- 扣款账户余额不足提醒

### 4. 公式虽有局部能力，但不可审计
当前可以填年化利率、总还款、总期数并进行部分推断，但还不够严谨。

缺少：

- 等额本息明确公式说明
- 等额本金明确公式说明
- 分期手续费公式说明
- 提前还款后的重算机制
- 每笔金额的“计算依据可见”

### 5. 日期监控还不够完整
当前虽有 `billDay` / `repaymentDay`，但还不够支撑严谨监控。

缺少：

- 提醒窗口配置（T-7 / T-3 / T-1）
- 逾期状态
- 宽限期
- 是否已生成当期计划
- 本期已还 / 未还状态

---

## 三、建议的数据结构分层

后续建议按 4 层来重构还款管理：

### A. 借贷主体层（Debt Master）
描述一笔贷款/负债本身。

建议字段：

- 借贷 ID
- 名称
- 类型
- 放款金额/授信额度
- 当前剩余本金
- 年化利率/手续费率
- 还款方式（等额本息 / 等额本金 / 分期 / 最低还款 / 自定义）
- 账单日
- 还款日
- 起始日期
- 总期数
- 默认扣款账户
- 状态（进行中 / 结清 / 逾期 / 冻结）

### B. 计划层（Repayment Schedule）
描述未来/历史每一期应还计划。

建议字段：

- 计划 ID
- 借贷 ID
- 期次
- 计划还款日
- 应还本金
- 应还利息
- 应还手续费
- 应还总额
- 剩余本金（期后）
- 计划状态（待还 / 部分还 / 已还 / 逾期）
- 计算快照/公式来源

### C. 记录层（Repayment Record）
记录真实发生的还款行为。

建议字段：

- 记录 ID
- 借贷 ID
- 计划 ID（可选）
- 实际还款时间
- 实际还款金额
- 实际扣款账户 ID
- 关联交易 ID
- 还款方式（手动 / 自动识别 / 自动扣款）
- 备注
- 是否撤销

### D. 校验层（Validation / Audit）
保障严谨性的对账与审计信息。

建议能力：

- 应还 vs 实还差异检查
- 少还 / 多还 / 漏还识别
- 提前还款影响提示
- 扣款账户异常提醒
- 公式计算与结果校验

---

## 四、建议的第一页改造方向

第一轮不建议直接上复杂引擎，建议先做“信息结构升级”。

### Phase 1（建议先做）
1. 补齐借贷主体字段
2. 明确默认扣款账户
3. 在页面上拆出：
   - 借贷主体信息
   - 近期还款计划
   - 实际还款记录
4. 为后续公式和计划表留出结构位置

这一步的目标不是一次做完所有功能，而是：

> 先把页面从“债务建议页”升级为“还款管理台账雏形”。

---

## 五、结论

当前 `RepaymentManagementPage` 已具备：

- 债务录入基础
- 部分利率/期数推断
- AI 建议与优先级分析

但距离“严谨的还款管理”还差三件关键事：

1. **逐期计划**
2. **真实还款记录**
3. **扣款账户闭环**

后续所有优化，建议围绕这三件事展开。
